# FastAPI核心概念复习

## 1. 依赖注入（Depends）到底做了什么？

我用一个比喻来解释：
- 就像公司前台，每个访客进来都要登记身份证
- `Depends(get_db)` 就是前台这个角色
- 每个需要数据库的接口，FastAPI会自动调用get_db，把数据库连接"注入"进去
- 这样做的好处：代码复用、便于测试

## 2. 路径参数 vs 查询参数

| 对比项 | 路径参数 | 查询参数 |
|--------|----------|----------|
| URL形式 | /tasks/5 | /tasks?task_id=5 |
| 是否必填 | 必填 | 可选 |
| 用途 | 定位唯一资源 | 筛选/排序/分页 |
| 示例 | 获取id=5的任务 | 获取已完成的任务列表 |

## 3. Pydantic的验证时机

请求进来时：
1. FastAPI接收JSON
2. 根据类型提示找到对应的Pydantic模型
3. Pydantic逐字段验证
4. 验证通过才进入函数体
5. 验证失败直接返回422

这意味着：你的函数体内拿到的数据**一定是合法的**。

## 4.请求处理流程图
```
浏览器请求 POST /tasks {"title": "学习"}
        ↓
    FastAPI路由匹配
        ↓
    找到对应的函数 create_task
        ↓
    检查函数参数：
    - task_data: TaskCreate → 用Pydantic验证请求体
    - db: Session = Depends(get_db) → 调用get_db获取连接
    - current_user = Depends(get_current_user) → 从Token解析用户
        ↓
    三个依赖都准备好后
        ↓
    执行函数体
        ↓
    返回响应（自动转JSON）
```

## FastAPI 面试核心问答

### 1. 如果一个请求没带Token，会在哪一步被拦截？
**答：在依赖注入阶段拦截，路由函数执行之前。**
Token 校验逻辑封装在 Depends 依赖内部，FastAPI 优先解析所有依赖参数。
缺少 Token、Token 格式错误或者 Token 失效时，直接抛出 HTTPException，不会进入路由内部执行业务代码。

### 2. response_model 和 return 的数据类型不匹配会发生什么？
**答：response_model 会对返回值进行校验、序列化、过滤处理，区分多种场景：**
- 缺少必填字段、单体对象与列表结构混淆：抛出 ValidationError，返回500错误；
- 字段类型不一致：默认模式下Pydantic尝试自动转型；开启 strict 严格模式则直接报错；
- 返回存在模型未定义的多余字段：不会报错，自动剔除多余字段，实现敏感信息脱敏；
- 直接返回 JSONResponse / Response 对象：response_model 完全失效，不再执行校验、过滤、类型转换。

### 3. 为什么要用 Depends(get_db) 而不是直接在函数里 db = SessionLocal()？
**答：主要四点优势：**
1. **会话生命周期自动管控**
借助 yield + finally，无论请求正常返回还是抛出异常，会话都会被关闭，防止数据库连接泄漏；手动创建会话容易在异常分支忘记close。

2. **单次请求共享同一个Session**
一次HTTP请求内多处注入该依赖只会创建一个会话，整条调用链路共用，保障事务一致性；多处手动创建 SessionLocal 会生成多个独立会话，事务割裂，数据互相不可见。

3. **便于统一维护扩展**
数据库会话的公共逻辑只需要修改 get_db 一处；硬编码方式需要多处复制修改，维护成本高。

4. **方便编写单元测试**
依赖注入支持Mock数据库会话，测试可不连接真实数据库；函数内部硬编码创建会话，依赖无法外部替换，单元测试难以实现。

> 补充：手动 SessionLocal() 仅适合脚本、后台任务；HTTP接口不推荐直接创建会话。
